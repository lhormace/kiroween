1.目的（Purpose）
何を達成したいのか。成功の定義は何か。
★→健康管理を容易かつ簡便に行いたい

2.背景（Background）
なぜやるのか。現状の課題は何か。
★→入力が難しくとても硬いシステムばかりなので、気軽にx.comに投稿する感じで、話し相手になってもらう感じのツールであれば、誰からも受け入れてもらいやすいと思った

3.スコープ（Scope）
含める範囲／含めない範囲。
★→健康管理を含める。それ以外は含めない。

4.利用者（User / Persona）
誰が使うのか。どんな行動をするのか。
★→私が使う。また、私と同じ価値観を持つ人が使う

5.機能要件（Functional Requirements）
どんな機能を持つべきか。
★→ユーザとシステムが対話する。
　最大140文字程度の文章を毎回受け付ける。
　対話の中で体重と体脂肪を入力する。
　対話の中でそのほかのコメントとして食べたもの、運動内容、困っていること、悩み相談を入力する。
　入力を受け付けた情報を元に、今後のプランをアイデアとして提示する
　また、入力した体重や体脂肪は、グラフとして表示できる。
　グラフは、１ヶ月、３ヶ月、半年の尺度で確認ができる
　食べたものの文章記述から自動で栄養素を推定し、今日の総摂取カロリーと PFC バランスを算出する
　
6.非機能要件（Non-Functional Requirements）
セキュリティ、性能、可用性、拡張性、監視、ログなど。
★→簡単にカスタマイズが可能であること。スクラッチ開発に適したアーキテクチャであること

7.制約条件（Constraints）
使う技術、予算、納期、組織ルール。
★→•バックエンド：SpringBoot（Java 21）
•ローカル DB：SQLite（JDBC 経由）
•グラフ生成：Java + JFreeChart またはフロント側で Chart.js
•対話ロジック：AWS Lambda（Java）＋ API Gateway
•MCP を用いて、最新の栄養学・睡眠研究・運動生理学の情報を
　対話の裏側で自動参照する“健康アドバイザーAI”として機能させる

8.入力と出力（Input/Output）
何を受け取り、何を返すものなのか。
★→入力：体重の数値、体脂肪の数値、その日食べたもの、運動内容
　出力：体重と体脂肪のグラフ。健康のためのコンサルティング結果
　　　　今日の総摂取カロリー
　　　　PFCバランス
　　　　短歌を生成し日記に記録する
　　　　対話内容を心理的読み取るエージェントが裏で動き、メンタル状態を勝手に推定する

9.関係する AWS サービス（もし AWS なら）
★→Lambda, CDK, StepFunctions, S3, API Gateway, IAM

10.成功基準（Acceptance Criteria）
★→実用性があると見做せる
